# ADMIN: 削除請求受付 SOP（管理者向け運用ドキュメント）

**目的**: ユーザーから削除請求を受けた際の対応手順を標準化する。
**対象**: 本サービスの運営者（Aki および将来的な追加運営者）
**SLA**: 削除請求受領から **14 日以内**（プライバシーポリシー §5-2 の確約）
**適用範囲**: Supabase 上の投稿者情報および関連データの一括削除

⚠️ **このドキュメントはリポジトリで管理されますが、ユーザー向けには公開しません。**
（route として置かないため、Vercel ビルドにも含まれません。GitHub リポジトリの可視性に依存します。）

⚠️ **具体的な SQL クエリは本ファイルに含めません。** リポジトリが public のため、user_id 単位で全テーブルを掃除する具体的な手順を直接公開しないよう、ローカル管理ファイル `.admin-sop-private.md`（`.gitignore` 対象）に分離してあります。本 SOP の章番号と `.admin-sop-private.md` の章番号は対応しています（§3 / §4 / §6）。

## 2. 依頼者本人確認

### 2.1 確認すべき項目

| 項目 | 必須 | 確認方法 |
|---|---|---|
| Twitch ユーザー名（login） | ✅ | 自己申告 + 連絡元のアカウント名と一致するか確認 |
| 削除範囲 | ✅ | 自己申告（「全データ」または「特定の投稿のみ」） |
| 連絡元アカウントの真正性 | ✅ | X / Twitch のプロフィールが Twitch login と紐付くか目視確認 |

### 2.2 なりすまし対策

- X DM の場合: 連絡元 X アカウントのプロフィールに **Twitch login** が掲載されているか確認
- Twitch Whispers の場合: 連絡元 Twitch アカウント = 申告 login で**自動的に一致**するため強い証跡
- 不審な場合: 「Twitch のログインで連絡をお願いします」と Twitch Whispers ルートに切り替え依頼

## 3. user_id の逆引き（手順）

Supabase に保存されているのは `user_id`（数値文字列）であり、ユーザーから受け取るのは Twitch login（人間可読名）。**最初に user_id を逆引き**する必要がある。

### 3.1 第一経路：Supabase 内のレコードから逆引き

依頼者が過去に投稿系の操作（テンプレート / カスタムアニメ）を行っていれば、Supabase ダッシュボードの SQL Editor で **対象テーブルから user_login を条件に SELECT** し、対応する user_id を取得する。

具体クエリは `.admin-sop-private.md §3.1` 参照。

### 3.2 第二経路：Twitch API 経由（Supabase に投稿レコードがない場合）

依頼者がいいね・通報・AI 生成しか行っていない場合、Supabase 内のレコードに user_login が記録されていないため、外部 API での逆引きが必要。

- Twitch の Helix API (`/helix/users?login=...`) を使う
- App access token を一時取得して GET リクエスト
- レスポンスから user_id（数値文字列）を取得

具体的な curl コマンド（client_secret を含む）は `.admin-sop-private.md §3.2` 参照。

### 3.3 user_id 確定後の件数確認

得られた user_id について、対象 6 テーブルにそれぞれ何件のレコードがあるかを件数確認する。合計 0 件なら「削除対象なし」として終了。

具体クエリは `.admin-sop-private.md §3.3` 参照。

## 4. 削除実行（手順）

### 4.1 全データ一括削除

依頼者が「全データ削除」を希望した場合、以下の順序で 6 テーブルから当該 user_id 紐付きのレコードを DELETE する。

削除順序の原則：**FK 参照される側を後に削除**する。具体的には:

1. `ai_animation_logs`
2. `animation_reports`
3. `animation_likes`
4. `likes`
5. `custom_animations`（CASCADE で `animation_likes` / `animation_reports` の関連分は連動削除）
6. `templates`（CASCADE 設定が確実でない場合は事前に `likes` を別途削除）

⚠️ **必ずトランザクション内（BEGIN ... COMMIT）で実行**してください。途中失敗時の部分削除を防ぐため。

具体 SQL は `.admin-sop-private.md §4.1` 参照。

### 4.2 削除後の確認

§3.3 と同じ件数確認クエリを再実行し、全テーブルで 0 件になっていることを目視確認する。

具体 SQL は `.admin-sop-private.md §4.2` 参照。

### 4.3 部分削除（特定の投稿のみ）

依頼者が「特定のテンプレートまたはカスタムアニメだけ消したい」と希望した場合、`id` 指定で個別 DELETE を行う。

注意点：
- 必ず `WHERE id = '...' AND user_id = 'USER_ID'` のように **user_id 条件を併用**して、誤削除を防ぐ
- カスタムアニメの場合、CASCADE で関連 `animation_likes` / `animation_reports` も連動削除される
- テンプレートの場合は `likes` の事前削除も必要（CASCADE 適用前なら）

依頼者が ID を知らない場合、本人投稿の一覧（タイトル / アニメ名 / 作成日）を見せて選んでもらう。

具体 SQL は `.admin-sop-private.md §4.3` 参照。

## 6. 削除実行ログの記録

⚠️ **削除実行は不可逆な操作のため、必ず記録を残すこと**。

### 6.1 ログファイルの場所

ローカル管理（リポジトリにコミットしない、SOP の手元実施記録）:

```
~/Desktop/Twitchエモート作成-admin-logs/deletion-log-YYYY.md
```

または Aki 個人の Notion / Obsidian / Google ドキュメント等。

### 6.2 ログ記録項目

カラム定義は `.admin-sop-private.md §6` 参照。受領日時・受領経路・依頼者の Twitch login・確認した user_id・削除範囲・削除実行日時・削除前後の件数・完了連絡日時・SLA 達成可否を記録する。

### 6.3 統計用集計

四半期ごとに「削除請求件数」「平均応答日数」を集計してプライバシーポリシーの実効性を確認。問題があれば SOP 改訂。

## 7. エッジケース対応

### 7.1 Twitch アカウント自体が存在しない / BAN された場合

- §3.2 の Twitch API ルートが 404 を返す
- → 過去に正常な login でデータが入っている可能性は残るので、§3.1 の Supabase 内逆引きを優先
- 削除可能な範囲で実施し、依頼者にその旨を返信

### 7.2 削除中にユーザーが新規投稿してきた場合

- §4.1 の DELETE 実行前後でタイムスタンプを記録
- 実行前に投稿された分のみ削除
- 実行後の新規投稿は別途依頼を受ける形で再実施
- 二度手間を避けるため、§4 実行直後に「ログアウトをお願いします」と明示

### 7.3 個別投稿削除と全削除の混在依頼

- 「特定の 1 件は残したい、それ以外を全削除」のような依頼
- §4.3 の部分削除を最初に行い、次に §4.1 の WHERE 条件に「残す ID 以外」の条件を加える
- 慎重に SQL を組み立てる、不安なら依頼者に「これでよろしいですか」と確認

### 7.4 第三者になりすました削除依頼の疑い

- §2.2 の本人確認で疑義がある場合、Twitch Whispers ルートに誘導
- それでも疑わしい場合、依頼を保留して Aki が個別判断
- なりすましによる誤削除を避けるため、確信が持てるまで実行しない

### 7.5 SLA（14 日）を過ぎそうな場合

- 受領後 7 日経過した時点で進捗確認の中間連絡を入れる
- 個人運営のため遅延の可能性を依頼者に伝え、誠実に対応
- 重大な遅延が発生した場合、原因と対応予定を率直に共有
